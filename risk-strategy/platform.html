推理轨迹示例

巡检 Agent 真实 ReAct 循环（3 轮摘要）
Round 1 · Observe → 发现 M1 回收率异常 ▼
Observe 调用 get_data_overview，获取当日全量指标快照
Think M1 回收率从 54% 降至 47%，环比 -7pp，超过历史 2σ 阈值。判断为显著异动，需深入拆解。
Act 调用 query_repay_trend（M1 回收率趋势）+ detect_anomaly（异动检测）
Round 2 · 归因 → 催员触达率下降 ▼
Observe 异动检测返回：S1 阶段 Large 案件回收率 MoM -2.8pp
Think 需拆解过程指标——催员触达率、案件分配量是否同步变化？如果触达率未降，可能是客群结构变化。
Act 调用 query_process_metrics（催员/触达/案量过程指标）
Round 3 · 确认 → 生成归因报告 ▼
Observe 触达率从 72% 降至 58%（环比 -14pp），案件分配量未变。确认瓶颈在触达环节。
Think 归因链完整：M1 回收率下降 ← 催员触达率下降 ← 需检查排班/线路容量。可生成最终报告。
Act 调用 generate_report，产出结构化 HTML 归因报告

Flywheel 学习机制 (Python) ▼ 
# Flywheel: 还款结果回流 → 策略权重更新
def update_strategy_weights(decision_log, repay_result):
    """决策日志 + 还款结果 → 策略库权重优化"""
    for case in decision_log:
        actual = repay_result.get(case["case_id"])
        if actual is None:
            continue
        # 比较 Agent 推荐 vs 实际结果
        reward = compute_reward(case["action"], actual)
        strategy_db.update_weight(
            strategy_id=case["strategy_id"],
            delta=LEARNING_RATE * reward
        )

5 种 Agent Workflow 模式

来自 Anthropic 官方最佳实践 · 每种均有实际业务映射
Prompt Chaining
将复杂任务分解为多个顺序步骤，每步输出作为下步输入，在步骤间加入质量门控。
实际应用 → 预估 Pipeline：Step01 → Step02 → ... → Step07a，7 步串联
Routing
对输入进行分类，将不同类型请求路由到对应的专业处理模块。
实际应用 → 巡检 Agent 工具选择：根据异动类型选择 DPD 趋势 / 回收趋势 / 过程指标工具
Parallelization
将任务拆分为可独立执行的子任务，并行处理后汇聚结果。
实际应用 → 多业务线并行分析：CL / BNPL / PPL 同时执行巡检，合并输出报告
Orchestrator-Workers
中央编排器动态分配任务给工作者，管理复杂多步骤流程。
实际应用 → AI 催收 Agent：Opus 编排推理 + Sonnet 执行转写分析
Evaluator-Optimizer
一个 LLM 生成内容，另一个 LLM 评估质量，形成迭代优化循环。
实际应用 → 巡检 Agent 双 LLM 互评：归因 LLM + 评估 LLM，矛盾点上报人工专家

五维评估检查

每次开发/交付前必须过一遍的质量门控
#维度一句话检查实际应用示例
1 人工介入率 上线后多少比例仍需人工干预？如何降低？ 巡检 Agent 人工复核率 <20%
2 鲁棒性 数据缺失/格式异常/上游延迟时，系统会怎样？ 覆盖率 <50% 不引用 DPD30，graceful 降级
3 业务可解释性 输出结果业务方能直接理解和使用吗？ 归因报告附带数据链：指标 → 数字 → 原因
4 可复制性 换一个业务线/产品，能否快速复用？ 巡检 Agent 覆盖 7 条业务线，配置化切换
5 决策一致性 同样输入跑两次，结果是否确定性一致？ 预估 Pipeline 纯确定性计算（6-3-1 加权）
