CHAPTER 05 · EVALUATION & MONITORING

评估与监控

训练只是工程的起点，评估才是真正的难题——没有标准答案、主观偏好难量化、长尾分布破坏 100% 信任。 这一章把我在 mvp-airobot 和贷后风控里搞清楚的东西讲给你：Golden Set 怎么建、LLM-as-Judge 哪些坑、教练-选手闭环为什么是蒸馏的工程变种、灰度阶梯每档卡什么、漂移监控分几类、以及一个常被业务方质疑的硬骨头——因果推断 vs 选择偏差。

⏱ 预计阅读 18 分钟 📅 2026-05-18 首发 ✍️ Mr. Yuan

0概览 · 为什么评估比训练更难

⏱ 2 分钟

很多团队把 80% 精力花在训练、20% 花在评估。结果上线后 bad case 像漏水的屋顶，永远在打补丁。这个比例是反的——LLM 工程的真正护城河，从来不是"用了哪个模型"，而是有没有一套能让模型持续变好的评估闭环。

评估和传统机器学习不同。传统 ML 有 ground truth、有 accuracy / F1 等收敛指标；LLM 面对的是三大结构性难题：

难题表现为什么这是死结
没有标准答案开放式生成（摘要 / 对话 / 报告），同一输入有 100 种合理输出传统 accuracy / F1 失效
主观偏好难量化"这个回答更专业" / "这个表达更人性化"没有 ground truth，标注者间一致性差
长尾分布头部样本 95% 正确，5% 长尾决定用户体验平均指标看不出问题，但单个翻车就上热搜
类比
训练 LLM 像教学生考试——有标准教材；评估 LLM 像当高考阅卷老师——面对 1000 份作文判到底哪份"立意更高"，且每份判完还要解释为什么。后者比前者难得多。

但难归难，是有套路的。一套能落地的评估闭环就 5 步：

图 0-1 · 评估不是"跑一次指标"，是 bad case 持续回流的飞轮

这五步对应本章 Section 1-4。Section 5 讲漂移监控，Section 6 讲因果推断——后两个是上线后才会真正暴露的硬骨头。

业务联系 · mvp-airobot

这套闭环不是空想——它就是 mvp-airobot 教练-选手模式的方法论骨架。Coach（Claude）做的就是 ① ② ③，Gate 做的是 ④ ⑤。这 5 步映射到 4 个 LLM 角色，是 Section 3 要展开的话题。

1Golden Set 设计

⏱ 3 分钟

Golden Set = 一组人工标注、业务方拍板过的高质量测试样本。每次改 prompt / 改模型 / 改数据，都用它跑一遍，分数不回退才能上线。这是评估闭环里第 ④ 步的核心资产——也是 90% 团队最容易偷懒的地方。

1.1 我搞清楚的几件事

维度建议反模式
规模100-300 条 起步10 条（覆盖不够）/ 10000 条（标注质量塌方）
场景覆盖高/中/低风险 + 头/腰/尾 9 宫格，每格 10-30 条只抓常见 case，长尾盲区
标注2-3 人独立标注，看 Cohen's Kappa > 0.6 才可信1 人标完直接上
版本v1 / v2 / v3 持续演进，旧版要保留做回归覆盖式更新，丢失基线

Meta 的 LIMA（Zhou et al., 2023）证明了一个反直觉的事：1K 条高质量样本能超过 10K 条噪声大杂烩。这条规律在 Golden Set 上同样成立——质量永远比数量重要。

类比
Golden Set 像医院的标准化体检套餐——不需要 10000 项全身扫描，但一套精心设计的 200 项必须覆盖所有关键器官（高/中/低风险全场景）。

1.2 字段设计（最少 5 个）

  • input：原始输入
  • expected_intent：业务意图（语义层，不一定字面答案）
  • recommended_seed：哪版回复更适合做种子（human / llm / either）
  • human_quality：人工标注质量（high / medium / low）
  • verdict：accept / review / reject
业务联系 · mvp-airobot 双轨交叉质检

我们在菲律宾贷后 AI 虚拟坐席项目里做过一次1010 条 Golden Set 建设：22 个 CSV × 抽样 + 全量混合，用 Gemini 2.5 Flash 原生音频做 12 维判定，得到 accept 84.2% / review 11.7% / reject 4.1% 的三档分流。

最反直觉的发现：recommended_seed 字段里 either 75.6% / llm 17.4% / human 6.4%——大部分场景人工标注和 LLM 标注语义等价，但有 17% 的 case LLM 转录比人工更准（人工标注常含"magaamount"/"seventen"这种连写错）。这之后我对"人工标注 = 金标准"这个假设就不再无条件接受了。

2LLM-as-Judge 范式

⏱ 3 分钟

LLM-as-Judge = 用强模型评弱模型。MT-Bench（Zheng et al., NeurIPS 2023）证明了 GPT-4 当 Judge 时，与人工标注一致率 > 80%——这意味着大规模评估可以自动化，但前提是要知道怎么绕开 LLM Judge 的三大坑。

2.1 三类评估打分模式

模式形式什么时候用
Pointwise单个回答打绝对分 1-10简单任务，但绝对分稳定性差
Pairwise两个回答比较 A/B/Tie主流，Chatbot Arena ELO 就是这套
Reference-based对照人工答案打分有标准答案场景

2.2 三大 Bias（必须规避）

Bias表现怎么规避
Position Bias判 A vs B 时，先出现的更容易被选中随机化顺序，或两个顺序都跑取均值
Verbosity Bias长回答更容易得高分（即使质量没更高）评分指令明确"控制长度因素"，或加 length penalty
Self-Preference Bias模型偏爱自己风格（GPT-4 偏 GPT-4，Claude 偏 Claude）跨模型交叉验证，至少用 2 个不同家族的 Judge
类比
LLM-as-Judge 像请名厨给餐厅打分——他能判断高低，但有自己的偏见：先尝的菜印象更深（Position Bias）、份量足的菜感觉更值（Verbosity Bias）、自己派系的菜更顺口（Self-Preference）。规避这些偏见，名厨才是好评委。

2.3 一种 3 层评分体系

这是我在贷后业务里逐步搞清楚的一套结构——把合规 / 业务 / 沟通三层分开，避免业务质量被沟通质量冲淡。它和 COPC 呼叫中心标准、催收行业 5 支柱可以无缝对应。

图 2-1 · 3 层 6 维度评分金字塔：合规一票否决 + 业务为本 + 沟通收尾
业务联系 · 1153 通对话评分

这套 3 层体系跑过 mvp-airobot 千通量级真实通话评分：L1 合规通过率 99.9%（仅 1 通违规）/ 质量均分 56.0 / 100 / 高质量占比 51.7%。整个评分用 Claude Code 订阅内的 Opus 4 跑 6 个并行 agent，外部 API 成本 $0——同样的事走 Sonnet API 大概 $1.85，走 Haiku $0.49。这是"Solution Architect 的成本意识"在工程层的体现。

3自迭代闭环 · 教练-选手模式

⏱ 4 分钟 · 本章核心 · 蒸馏的工程变种

评估完之后怎么改？这是整个评估闭环里最容易被低估的一环。很多团队的答案是"加规则"，结果 prompt 越加越长、token 爆、规则相互打架。

正确做法叫 Coach-Runner（教练-选手）模式——它本质是第 01 章蒸馏的工程变种，学术名叫 Prompt Distillation（行为蒸馏）。

3.1 两种反模式（先排除错的）

反模式 ① · 让弱 LLM 自迭代 prompt

"让 qwen-flash 看自己的 bad case，自己改 prompt" —— 越改越烂。我们试过：弱 LLM 看不到自己的盲点（让不会做的人判断"我哪里错了"），加规则只会堆 token、规则互相打架，缺乏"跳出当前局"的能力。

反模式 ② · 死规则模板贴贴

"建 17 个预定义失败模板，匹配上就套" —— 2 轮就 PLATEAU。模板穷举不完，不能处理新 case，跨 case 的模式识别也做不了。

3.2 正模式 · 4 角色分工

类比
学徒制——师傅（强 LLM）看徒弟（弱 LLM）干活，找问题、改方法、再让徒弟试。徒弟不评自己（他看不出问题），师傅也不亲自干活（他的角色是教练）。
图 3-1 · 4 角色闭环：Coach 不亲自跑 · Runner 不评自己 · Judge 中立 · Gate 兜底

3.3 4 条不可妥协的约束

  1. Coach 不重写整个 prompt，只 patch 局部段落（如 GUARDRAILS / LISTEN RULES）
  2. 每条新规则带 case ID 注释：# fix EXT_xxx_001 可追溯
  3. token cap 硬约束：避免 prompt 无限膨胀（我们设的 2700 cap）
  4. 保留人类调优过的 ROLE / GOAL / CORE FLOW 不动

3.4 vs RLHF / Fine-tuning

维度Coach-RunnerRLHFFine-tuning
上线周期即时周-月月-季
数据需求10-50 case turn 级1000+ 标注10K+ 标注
成本低（API）高（人工标注）极高（GPU 训练）
灵活性高（改 prompt 即生效）中低（模型 freeze）
可解释性高（prompt 可读）低低

→ 创业 / 中小公司 / 快速验证：先用 Coach-Runner 跑通业务，规模化再考虑 fine-tuning。

业务联系 · mvp-airobot v1 → v2

2026-05-11 在 mvp-airobot 落地这套闭环。Coach（Claude）看 4 NEVER FIXED + 2 BROKEN 的 turn-by-turn 数据写精准规则，v1 prompt 误报率 38%（把"标点/大小写差异"判为 semantic_mismatch）→ 业务侧抽样 50 条审 → Coach 修正 → v2 误报率降到 3%（-92%）。

整个闭环不到 1 周。如果走传统 fine-tuning 路径，光数据标注就要 1-2 个月。这就是 API LLM 时代蒸馏的工程变种的真实威力——蒸不了 logits，但能蒸 prompt 和数据。

回顾 · 这是 01 章蒸馏的工程版

在 第 01 章蒸馏 讲过：Coach-Runner 把强模型的判断力通过 prompt 工程而非权重训练 distill 到便宜模型上——所以学术名叫 Prompt 蒸馏 / 行为蒸馏。理论上是硬蒸馏的工程变种，实践上是当前生态最实用的能力下放路径。

4A/B 灰度阶梯

⏱ 2 分钟

评估通过的 prompt / 模型不是直接 100% 上线，要走阶梯灰度。Microsoft Kohavi 团队（KDD 2009 起一系列论文）把这套思路系统化，后被 Netflix / Google / Meta 工业化——核心是每一档独立卡指标，回退立刻退档。

类比
阶梯灰度像药物临床试验三期——I 期小范围（5%）查安全性，II 期中等规模（20%）看疗效，III 期大规模（50%）确认长尾不良反应，最后才上市（100%）。每期都可能因为意外撤回。
图 4-1 · 5 → 20 → 50 → 100% 阶梯 · 每档独立卡指标

4.1 每档怎么卡

阶梯主要看什么退档触发
5%显式 bad case / 投诉 / 合规告警1 例严重违规 → 立刻退
20%指标统计显著性（z-test / chi-square / t-test）主指标置信下降 → 退
50%长尾 case 暴露 + 业务指标真实变化累计 bad case 超阈值 → 退
100%漂移监控 + 下一轮迭代漂移告警 → 触发下一轮 Coach
业务联系 · 菲律宾 5/20 灰度

2026-05-20 启动菲律宾贷后全 bucket（S0/S1/S2/M1）AI 虚拟坐席灰度。金融场景比互联网更严——合规违规是一票否决，1 例严重违规可能就要回退到 0%，不只是退一档。所以 5% 起步阶段我们看的不是"指标好不好"，是"有没有一次合规出格"。

5漂移监控

⏱ 2 分钟

模型上线不是终点。线上数据分布会漂移——可能是季节性、用户行为变化、上游链路变化。统计学上分三类：

5.1 三类漂移

类型学术名表现应对
输入漂移Covariate ShiftP(X) 变了，P(Y|X) 不变触发 Golden Set 扩样
输出漂移Label ShiftP(Y) 变了跑 Judge 重评，对照基线
概念漂移Concept DriftP(Y|X) 变了启动归因 + 重训 / 改 prompt

5.2 怎么检测

金融业界最常用的是 PSI（Population Stability Index）——分箱后看相对熵，PSI > 0.25 = 显著漂移。LLM 场景更倾向用 embedding 距离 + 业务指标双指标联动。

方法适用
KS test单变量连续分布
PSI金融风控传统手段，分箱式
Embedding 距离文本输入用余弦相似度对比基线
业务指标联动技术 + 业务双指标，防单边误判
类比
漂移监控像体检追踪——同一个指标隔半年再测一次，对比基线。单次正常不代表健康，连续 3 次往同一方向偏移就要查原因。
业务联系 · SESSIONS_DIR 硬编码踩坑

真实工程里最常见的不是"漂移检测算法选哪个"，而是数据源扫描不全。我们的 Skill 自迭代日报曾经硬编码 SESSIONS_DIR = "C:/.../yuanpeng03"，只扫 C 盘 30 个旧 session，完全没扫 E 盘 111 个真实工作 session——这就是当时"感觉 Skill 没自迭代"的根因。

修复后 total_sessions: 0 → 57 / skills_active: 0 → 22/22 / skills_avg_hitrate: 0% → 15.3%。教训：多目录扫描必须用 discover 函数而不是硬编码路径。

6因果推断与选择偏差 ⭐

⏱ 3 分钟 · 整章最有差异化的部分

这是整章最容易被业务方质疑、也是最有方法论价值的部分。算法团队给一份"模型上线后回收率 +10pp"的报告，业务方会问：

业务方的灵魂拷问

"聊得好的客户本来就好聊，可能单纯是聊得好的客户特征就更优。你这 +10pp 是模型功劳，还是选择偏差（Selection Bias）？"

这个问题答不上来，整套评估闭环的可信度归零。它本质是一个因果推断（Causal Inference）问题。

6.1 三个概念必须分清

概念定义例子
相关性X 和 Y 同向变化冰淇淋销量和溺水率同时升高
因果性X 改变会导致 Y 改变夏天 → 冰淇淋销量↑ + 游泳人数↑ → 溺水↑
选择偏差样本不是随机的，自带分布差异只看"主动接电话的客户" → 这部分本来还款意愿就高

6.2 Simpson's Paradox · 整体和分组可以相反

Pearl & Mackenzie 在 The Book of Why（2018）里强调过：整体趋势和分组趋势可以完全相反。这是评估里最危险的陷阱。

视角看到的结论
整体高质量对话 → 高回收率（+10pp）"模型有效"
分组（按客户配合度）高配合客户里：模型差异 +5pp
低配合客户里：模型差异 +3pp		整体的 +10pp 部分来自样本结构差异
结论必须控制变量（分层 / 匹配），才能估出模型的纯净效应
类比
就像评估"运动 → 减肥"这个因果。整体看运动多的人体重轻——但可能是"本来瘦的人才喜欢运动"。只有控制初始体重、年龄、饮食后，运动的效应才能被可靠估计。

6.3 控制变量法 · 强因果区间

金融场景里通常做不了 RCT（不能随机分配客户给不同策略），实用方法是找一个"混淆变量可控"的子集，看效应是否还在：

  • 整体视角：高质量 vs 低质量 → 业务指标差异 A pp
  • 问题：包含"配合的客户对话天然更长" 这种混淆
  • 切到"对话长度可比"区间（如 4-6 轮）→ 业务指标差异 B pp
  • 判断：若 B ≥ A，效应不消失甚至放大 → 不是选择偏差，是真实因果
业务联系 · mvp-airobot 千通验证

这套方法在 mvp-airobot 千通量级真实通话中验证过：

• 整体 2×2：高质量 38.3% vs 低质量 27.8% · +10.4pp

• 强因果区间（4-6 turn）：控制配合程度后差异 +17.8pp（比整体更大！）

第二个数字让业务方从"质疑选择偏差"变成"接受模型有真实增益"。控制变量后效应反而放大，是因果性的最强证据——这套逻辑可以泛化到任何 LLM 上线效果归因。

为什么这一节独特

业界关于 LLM 评估的资料 90% 在讲"用什么 metric"，只有 10% 在讲"指标变化是不是真因果"。能区分这两件事，是 Solution Architect 和工程师的分水岭——也是真正能让评估闭环对业务方"立得住"的关键。
